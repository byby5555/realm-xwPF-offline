# Docker 化 + Web 界面配置 + 账户密码认证可行性评估（含旧版功能全量对齐）

## 结论

**可以实现旧版能力的“功能全覆盖”，并且可以在 Web 端做更清晰的可视化优化。**

但建议采用“**分阶段对齐 + 双轨兼容**”策略：
- 短期：先做到 100% 规则功能可用 + 核心运维动作可用
- 中期：补齐高级功能（故障转移、流量狗、链路测试、导入识别）
- 长期：在不牺牲能力的前提下完成交互优化与权限体系升级

## 你关心的核心问题：能不能“全部实现旧版本功能”？

### 答案：能，但要按模块拆解推进

现有项目已经具备明确的功能边界：
- 规则管理（新增/编辑/删除/启停）
- 配置生成（`config.json`）
- 服务控制（启动/重启/状态）
- 负载均衡、故障转移、导入导出、网络测试等扩展能力

这些能力本质上都可映射到：
1. 后端 API（配置与任务编排）
2. Web 前端（可视化交互）
3. 容器运行时（realm 进程 + sidecar 任务）

## Web 可视化优化建议（逻辑清晰）

### 1) 信息架构（IA）

建议左侧导航分为 6 个一级菜单：
1. **仪表盘**：服务状态、活动规则数、告警
2. **规则管理**：单规则 + 规则组 + 批量操作
3. **网络与安全**：TLS/WS/WSS、Proxy Protocol、MPTCP
4. **可用性策略**：负载均衡、故障转移、探活配置
5. **工具中心**：导入导出、OCR 导入、链路测试
6. **系统设置**：用户认证、审计日志、备份恢复

### 2) 关键交互优化

- **向导式创建规则**：基础参数 → 安全参数 → 高级参数 → 预览提交
- **配置差异预览（Diff）**：提交前显示变更点，降低误操作
- **一键回滚**：每次发布配置自动生成版本快照
- **批量操作**：多选启停、批量删除、批量打标签
- **状态可视化**：规则卡片显示“启用/禁用/异常/探活失败”

### 3) 可维护性优化

- 前后端字段统一 schema（避免同名不同义）
- 后端统一校验器（端口、IP、域名、证书路径、策略组合）
- 配置写入全链路原子化（临时文件 + 校验 + 替换 + reload）

## 推荐技术架构（面向全量功能）

### 运行层
- `realm-runtime`：仅负责 realm 进程和配置加载
- `realm-panel-api`：规则管理、认证、任务调度
- `realm-panel-web`：前端 UI
- `task-runner`（可选）：执行 OCR、测速、探活等耗时任务

### 数据层
- `/etc/realm`：规则文件与 `config.json`（卷）
- `/var/lib/realm-panel`：用户、会话、审计、任务记录（卷）

### 安全层
- 账号密码认证（bcrypt/argon2）
- Session/JWT（二选一，推荐短期 session）
- 登录限流、IP 黑名单、CSRF 防护
- HTTPS 终止（Nginx/Caddy/Traefik）

## 分阶段实施（保证“全功能可达”）

### Phase 1：核心对齐（必须）
- 规则 CRUD、启停、配置生成、realm 重载
- 登录认证与基础权限
- 规则列表/详情/编辑页

### Phase 2：高级能力（补齐旧版）
- 负载均衡、故障转移
- 导入导出、OCR 导入
- 审计日志与配置版本

### Phase 3：运维增强（超越旧版）
- 链路测试报告可视化
- 流量统计与告警中心
- 多角色权限（RBAC）与 2FA（可选）

## 风险与规避

1. **并发改配置导致冲突**
   - 规避：文件锁 + 乐观版本号 + 提交前 diff 检查

2. **容器网络模型差异（IPv6/host 模式）**
   - 规避：提供标准 compose 模板（bridge/host 双模板）

3. **高级功能任务耗时阻塞**
   - 规避：任务队列异步化 + 前端轮询/推送状态

4. **旧配置兼容性**
   - 规避：先兼容 `rule-*.conf`，再逐步迁移到结构化存储

## 一键安装拉取 Docker 命令（可直接落地）

你可以用和当前项目一致的分发思路：**把安装脚本放在仓库根目录，然后通过 GitHub Raw 一键执行**。

### 1) 标准一键安装命令（推荐）

```bash
curl -fsSL https://raw.githubusercontent.com/byby5555/realm-xwPF-offline/main/install-docker-panel.sh | sudo bash
```

### 2) 网络受限时的加速源命令

```bash
curl -fsSL https://v6.gh-proxy.org/https://raw.githubusercontent.com/byby5555/realm-xwPF-offline/main/install-docker-panel.sh | sudo bash
```

### 3) 命令背后建议完成的动作（脚本内）

`install-docker-panel.sh` 建议按以下流程执行：
1. 检测并安装 Docker / Docker Compose（若缺失）
2. 创建安装目录（如 `/opt/realm-panel`）
3. 下载或写入 `docker-compose.yml`
4. 执行 `docker compose pull`
5. 执行 `docker compose up -d`
6. 输出访问地址与初始化账号密码提示

示例核心命令：

```bash
cd /opt/realm-panel
docker compose pull
docker compose up -d
```

### 4) 若你希望“自动生成一键命令”

本质就是把脚本文件名和分支拼成 Raw URL：

```text
https://raw.githubusercontent.com/<owner>/<repo>/<branch>/<script>.sh
```

对应执行模板：

```bash
curl -fsSL <RAW_URL> | sudo bash
```

> 结论：先把安装逻辑固化到 `install-docker-panel.sh`，再发布上述 `curl -fsSL ... | sudo bash` 即可形成“一键安装拉取 Docker 并启动服务”的命令。
